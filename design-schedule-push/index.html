企业微信群发方案

Posted on 2020-02-21 | In 设计与架构 |

背景

目前老师用来和用户沟通的工具主要是企业微信，为了提升老师的工作效率，决定引入企微宝（类似 wehub）来帮助老师

此方案就是以企微宝为基础，实现企业微信自动群发

目标

  1. 支持老师给多个学生发送多条消息，我们称为一个群发计划
  2. 群发计划支持两种模式：定时计划（计划内每条消息的发送是固定时间） 和 立即执行计划（执行群发计划的时候立即发送第一条，之后每条消息依次发送）
  3. 老师能随时 暂停，执行，取消 群发计划
  4. 支持内容变量替换，比如替换${studentFullName}
  5. 一个学生多条消息只要有一个失败就需要停止这个学生剩余消息的发送或者停止这个群发计划的发送
  6. 群发计划执行结果支持通过学生维度和任务维度统计
  7. wehub 的群脚本功能可以迁移过来

表结构


CREATE TABLE plan_record
(
id bigint auto_increment NOT NULL primary key,
plan_id bigint NOT NULL comment ‘关联的计划’,
relation_id varchar(64) comment ‘关联的业务数据 ID’,
relation_type varchar(64) comment ‘关联的业务数据类型’,
status varchar(32) NOT NULL comment ‘状态’,
run_at timestamp NOT NULL comment ‘期望执行时间’,
operator_oid varchar(32) NOT NULL comment ‘操作者 oid’,
operator_name varchar(64) comment ‘操作者名字’,
ext varchar(2048) comment ‘额外记录’,
created_at timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP comment ‘创建时间’,
updated_at timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP comment ‘更新时间’
) COMMENT = ‘执行计划记录’;

CREATE TABLE task_record
(
id bigint auto_increment NOT NULL primary key,
plan_record_id bigint NOT NULL comment ‘关联的计划记录’,
task_id bigint NOT NULL comment ‘关联的任务’,
relation_id varchar(64) comment ‘关联的业务数据 ID’,
relation_type varchar(64) comment ‘关联的业务数据类型’,
job_id varchar(64) comment ‘企微宝服务返回的任务 id’,
status varchar(32) NOT NULL comment ‘状态’,
failed_type varchar(32) comment ‘失败类型’,
run_at timestamp NOT NULL comment ‘期望执行时间’,
push_at timestamp comment ‘提交到企微宝服务的时间’,
done_at timestamp comment ‘结束执行时间’,
operator_oid varchar(32) NOT NULL comment ‘操作者 oid’,
operator_name varchar(64) comment ‘操作者名字’,
ext varchar(2048) comment ‘额外记录’,
created_at timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP comment ‘创建时间’,
updated_at timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP comment ‘更新时间’
) COMMENT = ‘执行任务记录’;

业务流程举例说明

假设要实现 100 个学生立即发 5 条消息:

  1. 创建一个立即执行类型的 plan，category 为企业微信群消息
  2. 在这个 plan 下面创建好 5 条消息，设置好消息顺序以及间隔时间
  3. 在花名册点选好要发送的学生，开始执行 plan，这个时候 首先会创建 plan_record，relation_type为 student，status为RUNNING，run_at为当前时间
  4. 根据传入的 studentOids 和 plan_record 来创建 task_record，每个学生每个 task 会创建一个 task_record，也就是会创建 500 条记录。注意创建task_record的时候需要按一个学生创建完再创建另一个学生的顺序，创建同一个学生的多条 task 的时候需要按照 task 的 priority
  5. 定时任务，需要给任务加锁，找到所有 run_at 小于当前时间且状态为 RUNNING 的 plan_record，然后再找所有状态为 RUNNING 的 task_record，保证一个RUNNING状态下的 plan_record必须有一个 RUNNING 状态的task_record，如果没有，需要执行这个 plan_record 的下一条 task_record（run_at 小于当前时间的最小的 且 自增 id 最小的）；如果plan_record 下的所有 task_record 为结束状态（FIALED、SUCCESSED、TIMEOUT），则更新为 FINISHED
  6. 企微宝服务每一个 job 执行完毕会发事件，当前服务监听事件来更新 对应 job_id 的 task_record status 为 SUCCEEDED 或 FAILED，这个 plan_record 下面所有的 task_record 状态更新完毕，即可以更新 plan_record 的 status 为 FINISHED
  7. 前端可以按 plan_record的 ID 或者 relation_id（也就是 student_oid） 查询 task_record 的执行情况

Hint

目前导致发送失败的有三类原因：

  1. 老师未登录企微宝
  2. 学生没有加老师好友
  3. 企业微信自己发送问题

每个 plan_record 下的 task_record 的执行是串行，有且只有一个再 RUNNING 中的 task_record，一个 task_record 执行完毕后才能执行下一个

监控

  • plan_record 创建数量
  • task_record 创建数量
  • task_record 分类统计（SUCCEED、FAILED、CANCELED）
